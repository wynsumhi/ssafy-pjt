# 커밋 컨벤션

본 문서는 지구깡 프로젝트의 커밋 메시지 작성 규칙을 정의합니다.

## 커밋 메시지 구조

커밋 메시지는 다음 형식을 따릅니다.

```
<type>(<scope>): <subject>

[optional body]

[optional footer]
```

### 구성 요소

- **type**: 커밋의 유형 (필수)
- **scope**: 변경 범위 (선택)
- **subject**: 변경 내용 요약 (필수)
- **body**: 상세 설명 (선택)
- **footer**: 이슈 참조 및 기타 메타 정보 (선택)

## Type

커밋 유형은 다음 중 하나를 사용합니다. 자주 사용하는 순서대로 정렬했습니다.

| Type     | 설명                 | 예시                                      | 사용 빈도 |
| -------- | ------------------ | --------------------------------------- | ----- |
| feat     | 새로운 기능 추가          | feat(draw): AI 추천 알고리즘 구현               | ⭐⭐⭐   |
| fix      | 버그 수정              | fix(comment): 익명 댓글 순번 중복 버그 수정         | ⭐⭐⭐   |
| refactor | 코드 리팩토링 (기능 변화 없음) | refactor(articles): 게시글 쿼리 로직 함수 분리     | ⭐⭐    |
| docs     | 문서 수정              | docs(wiki): API 명세서 작성                  | ⭐⭐    |
| style    | 코드 포맷팅, 세미콜론 누락 등  | style(views): PEP 8 포맷 적용               | ⭐     |
| test     | 테스트 코드 추가 및 수정     | test(draw): AI 추천 알고리즘 테스트 추가           | ⭐     |
| chore    | 빌드 및 설정 변경         | chore(deps): Django 5.1.4 업데이트          | ⭐     |
| perf     | 성능 개선 (측정 가능한 최적화) | perf(articles): N+1 쿼리 해결로 조회 속도 85% 개선 | ⭐     |

### refactor vs perf 구분

두 타입 모두 코드를 개선하지만 목적이 다릅니다.

| 기준        | refactor                | perf                   |
| --------- | ----------------------- | ---------------------- |
| **주 목적**  | 코드 구조, 가독성, 유지보수성 향상    | 실행 속도, 메모리 사용량 최적화     |
| **기능 변화** | 없음                      | 없음                     |
| **성능 변화** | 없음 (또는 목적이 아님)          | 있음 (목적)                |
| **측정 필요** | 필요 없음                   | 필요함 (Before/After 비교)  |
| **예시**    | 함수 분리, 변수명 변경, 중복 코드 제거 | 쿼리 최적화, 알고리즘 개선, 캐싱 적용 |

**refactor 예시:**

```bash
# 코드를 더 명확하게 만들었지만 성능은 그대로
refactor(draw): 거리 계산 로직을 별도 함수로 추출
refactor(views): 200줄 뷰 함수를 여러 작은 함수로 분리
refactor(models): Article 모델 필드명 명확화
```

**perf 예시:**

```bash
# 성능 개선을 목표로 작업하고 측정 가능한 결과
perf(articles): select_related 적용으로 게시글 조회 85% 개선
perf(draw): Haversine 계산 캐싱으로 뽑기 속도 2배 향상
perf(api): 응답 시간 850ms → 120ms로 단축
```

**애매한 경우**: 리팩토링 과정에서 우연히 성능이 좋아졌다면 `refactor` 사용

## Scope

변경 범위를 명시합니다. Django 앱 이름, 모듈 또는 컴포넌트를 사용합니다.

### 사용 가능한 Scope (지구깡 프로젝트)

```
draw         # 뽑기 기능
auth         # 인증 및 권한
accounts     # 사용자 관련
articles     # 게시글 관련
comment      # 댓글 관련
places       # 장소 관련
like         # 좋아요 기능
follow       # 팔로우 기능
api          # API 엔드포인트
models       # 데이터 모델
views        # 뷰 로직
serializers  # 시리얼라이저
settings     # 설정 파일
wiki         # Wiki 문서
```

Scope는 생략 가능하지만, 변경 범위가 명확한 경우 명시하는 것을 권장합니다.

## Subject

변경 내용을 간결하게 요약합니다.

### 작성 규칙

- 명령형 현재 시제를 사용합니다.
- 첫 글자는 소문자로 작성합니다.
- 마침표를 붙이지 않습니다.
- 50자 이내로 작성합니다.

### 올바른 예시

```
feat(draw): AI 추천 알고리즘 구현
fix(comment): 익명 댓글 순번 중복 버그 수정
docs(wiki): API 명세서 작성
refactor(articles): 게시글 쿼리 로직 함수 분리
perf(api): select_related 적용으로 응답 속도 85% 개선
```

### 잘못된 예시

```
feat(draw): AI 추천 알고리즘을 구현했습니다.     # 과거형, 마침표 사용
fix: bug fix                                   # 설명 불명확
Update code                                    # type 누락, 대문자 시작
feat(draw): 사용자 좋아요 기록을 분석해서 코사인 유사도 계산  # 너무 장황함
```

## Body

상세한 설명이 필요한 경우 본문을 작성합니다.

### 작성 규칙

- Subject와 한 줄 띄우고 작성합니다.
- 72자마다 줄바꿈을 합니다.
- 무엇을, 왜 변경했는지 설명합니다.
- 어떻게 변경했는지는 코드로 확인할 수 있으므로 생략 가능합니다.

### 예시

```
feat(draw): AI 추천 알고리즘 구현

사용자의 좋아요 기록을 분석하여 선호도 기반으로
게시글을 추천하는 AI 모드를 구현했습니다.

- 최근 좋아요 20개의 임베딩 벡터 수집
- 코사인 유사도 계산
- 상위 20% 후보군에서 가중 랜덤 선택
```

## Footer

이슈 참조나 Breaking Change를 명시합니다.

### 이슈 참조

```
Closes #15
Resolves #23
Ref #45
```

### Breaking Change

```
BREAKING CHANGE: 뽑기 API 응답 형식이 변경되었습니다.
```

## 커밋 예시

### 기본 커밋

```
feat(draw): AI 추천 알고리즘 구현
```

### 상세 설명 포함

```
feat(draw): AI 추천 알고리즘 구현

사용자의 좋아요 기록을 분석하여 선호도 기반으로
게시글을 추천하는 AI 모드를 구현했습니다.

- 최근 좋아요 20개의 임베딩 벡터 수집
- 코사인 유사도 계산 (NumPy 사용)
- 상위 20% 후보군에서 가중 랜덤 선택
- 좋아요 데이터 부족 시 랜덤 모드로 폴백

Closes #15
```

### 버그 수정

```
fix(comment): 익명 댓글 순번 중복 버그 수정

get_anonymous_number() 함수에서 기존 사용자의 순번을 
먼저 찾도록 로직을 개선했습니다.
동일 게시글에 여러 사용자가 댓글 작성 시 순번이 
중복되는 문제를 해결했습니다.

Resolves #23
```

### 문서 작업

```
docs(wiki): API 명세서 작성

프론트엔드 연동을 위한 API 명세서를 작성했습니다.
전체 엔드포인트, 요청/응답 형식, 에러 코드를 포함합니다.
```

### 리팩토링

```
refactor(draw): 거리 계산 로직을 별도 함수로 추출

중복되던 Haversine 공식 계산 코드를 
calculate_distance() 함수로 분리하여
코드 재사용성과 가독성을 개선했습니다.
```

### 성능 개선

```
perf(articles): N+1 쿼리 해결로 조회 속도 85% 개선

게시글 목록 조회 시 select_related('author', 'place')와
prefetch_related('likes')를 적용하여
N+1 쿼리 문제를 해결했습니다.

측정 결과:
- 쿼리 수: 41개 → 3개
- 응답 시간: 850ms → 120ms (85.9% 개선)
```

### 설정 변경

```
chore(deps): Django 5.1.4 업데이트

Django를 5.1.3에서 5.1.4로 업그레이드하고
보안 패치를 적용했습니다.
```

## 커밋 작성 팁

### 의미 있는 단위로 커밋

하나의 커밋은 하나의 논리적 변경사항을 포함해야 합니다.

```bash
# 좋은 예: 기능별로 분리
git commit -m "feat(draw): Haversine 거리 계산 로직 구현"
git commit -m "test(draw): 거리 계산 테스트 추가"
git commit -m "feat(draw): 거리 필터링 API에 적용"

# 나쁜 예: 여러 변경사항을 한 번에
git commit -m "feat(draw): 뽑기 기능 전체 완성"
```

### 커밋 전 확인사항

```bash
# 변경된 파일 확인
git status

# 변경 내용 확인
git diff

# 스테이징된 내용 확인
git diff --staged
```

### 커밋 메시지 수정

마지막 커밋 메시지를 수정할 수 있습니다. (push 전에만 사용)

```bash
git commit --amend -m "fix(comment): 익명 댓글 순번 중복 버그 수정"
```

## 자주 하는 실수

### 불명확한 메시지

```bash
# 나쁜 예
git commit -m "update"
git commit -m "fix bug"
git commit -m "WIP"

# 좋은 예
git commit -m "feat(places): 장소 저장 기능 구현"
git commit -m "fix(comment): 익명 순번 중복 버그 수정"
git commit -m "docs(wiki): 설치 가이드 업데이트"
```

### Type과 내용 불일치

```bash
# 나쁜 예
git commit -m "feat: 버그 수정"        # feat인데 버그 수정?
git commit -m "fix: 새 기능 추가"      # fix인데 새 기능?

# 좋은 예
git commit -m "fix(draw): 중복 추천 방지 로직 추가"
git commit -m "feat(articles): 좋아요 기능 구현"
```

### 너무 많은 변경사항

```bash
# 나쁜 예
- 10개 파일 수정
- 새 기능 3개 추가
- 버그 5개 수정
  → 하나의 커밋으로 처리

# 좋은 예
각 변경사항을 논리적 단위로 분리하여 개별 커밋 생성
```

### refactor와 perf 혼동

```bash
# 나쁜 예
refactor(api): 쿼리 최적화로 응답 속도 85% 개선  # 성능 개선이 목적이면 perf

# 좋은 예
perf(api): select_related 적용으로 응답 속도 85% 개선
refactor(api): 뷰 함수를 여러 작은 함수로 분리
```

## 참고사항

- 커밋 메시지는 동료 개발자와 미래의 자신을 위한 것입니다.
- 명확하고 일관된 메시지는 프로젝트 이력 추적을 용이하게 합니다.
- 자동화 도구(Changelog 생성 등)를 사용할 때 일관된 형식이 중요합니다.
- 의미 있는 단위로 자주 커밋하는 것이 좋습니다.
- 실제 프로젝트 기능과 구조를 반영한 커밋 메시지를 작성합니다.
