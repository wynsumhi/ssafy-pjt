# 브랜치 컨벤션

본 문서는 지구깡 프로젝트의 브랜치 전략 및 명명 규칙을 정의합니다.

## 브랜치 구조

### 주요 브랜치

프로젝트는 두 개의 주요 브랜치를 유지합니다.

| 브랜치     | 용도           | 특징             |
| ------- | ------------ | -------------- |
| main    | 배포 가능한 안정 버전 | Direct push 금지 |
| develop | 개발 통합 브랜치    | Direct push 금지 |

## 브랜치 명명 규칙

브랜치 이름은 다음 형식을 따릅니다.

```
<type>/<issue-number>-<description>
```

### 구성 요소

- **type**: 작업 유형 (필수)
- **issue-number**: GitHub Issue 번호 (필수)
- **description**: 작업 내용 간단 설명 (필수)

### Type 목록

브랜치 타입은 커밋 컨벤션과 동일하게 사용합니다.

| Type     | 설명                 | 예시                           | 사용 빈도 |
| -------- | ------------------ | ---------------------------- | ----- |
| feat     | 새로운 기능 추가          | feat/15-ai-draw-algorithm    | ⭐⭐⭐   |
| fix      | 버그 수정              | fix/23-anonymous-comment-bug | ⭐⭐⭐   |
| refactor | 코드 리팩토링 (기능 변화 없음) | refactor/34-article-query    | ⭐⭐    |
| docs     | 문서 수정              | docs/10-api-specification    | ⭐⭐    |
| style    | 코드 포맷팅, 세미콜론 누락 등  | style/7-pep8-formatting      | ⭐     |
| test     | 테스트 코드 추가 및 수정     | test/19-draw-algorithm       | ⭐     |
| chore    | 빌드 및 설정 변경         | chore/5-django-update        | ⭐     |
| perf     | 성능 개선 (측정 가능한 최적화) | perf/28-query-optimization   | ⭐     |
| hotfix   | 운영 중 긴급 수정         | hotfix/99-jwt-token-error    | 🚨    |

### 규칙

- 영문 소문자만 사용합니다.
- 단어 구분은 하이픈(`-`)을 사용합니다.
- Issue 번호는 `#` 없이 숫자만 사용합니다.
- 설명은 구체적이고 간결하게 작성합니다.
- 개인 이름이나 날짜를 포함하지 않습니다.
- Issue가 없는 경우에만 `<type>/<description>` 형식 허용

### 올바른 예시

```
feat/15-ai-draw-algorithm
feat/23-follow-feed
fix/8-anonymous-comment-sequence
fix/12-jwt-token-refresh
refactor/34-article-query-optimization
docs/10-api-specification

test/19-draw-algorithm-test

```

### 잘못된 예시

```

fix/bug                          # Issue 번호 누락
feat/15_ai_algorithm             # 언더스코어 사용
FEAT/15-ALGORITHM                # 대문자 사용

```

## Issue와 브랜치 타입 매핑

Issue 템플릿과 브랜치 타입을 일관되게 사용합니다.

| Issue 템플릿       | 브랜치 타입   | 예시                                  |
| --------------- | -------- | ----------------------------------- |
| Feature Request | feat     | feat/15-ai-draw-algorithm           |
| Bug Report      | fix      | fix/23-anonymous-comment-bug        |
| Refactoring     | refactor | refactor/34-article-query           |
| Documentation   | docs     | docs/10-api-specification           |
| Chore           | chore    | chore/5-django-update               |
| (없음)            | style    | style/7-pep8-formatting             |
| (없음)            | test     | test/19-draw-algorithm              |
| (없음)            | perf     | perf/28-select-related-optimization |
| (긴급 수정)         | hotfix   | hotfix/99-session-timeout           |

## 브랜치 작업 절차

### 새 브랜치 생성

```bash
# develop 브랜치에서 최신 코드 받기
git checkout develop
git pull origin develop

# 새 작업 브랜치 생성 (Issue #15: AI 추천 알고리즘)
git checkout -b feat/15-ai-draw-algorithm
```

### 작업 진행

```bash
# 변경사항 확인
git status

# 파일 스테이징
git add .

# 커밋 (커밋 컨벤션 참고)
git commit -m "feat(draw): AI 추천 알고리즘 구현"
```

### 원격 저장소 푸시

```bash
# 원격 저장소에 브랜치 푸시
git push origin feat/15-ai-draw-algorithm
```

### Pull Request 생성

GitHub에서 Pull Request를 생성합니다. 자세한 내용은 Pull Request 가이드를 참고하시기 바랍니다.

## 브랜치 삭제

작업이 완료되어 `develop`에 병합된 브랜치는 삭제합니다.

```bash
# 로컬 브랜치 삭제
git branch -d feat/15-ai-draw-algorithm

# 원격 브랜치 삭제
git push origin --delete feat/15-ai-draw-algorithm
```

GitHub에서 Pull Request 병합 시 자동 삭제 옵션을 활성화하는 것을 권장합니다.

## Hotfix 절차

운영 중 긴급 수정이 필요한 경우에만 hotfix 브랜치를 사용합니다.

```bash
# main 브랜치에서 hotfix 브랜치 생성 (Issue #99)
git checkout main
git pull origin main
git checkout -b hotfix/99-session-timeout

# 수정 작업 및 커밋
git commit -m "fix(auth): 세션 타임아웃 오류 수정"

# main으로 Pull Request 생성 및 병합

# develop에도 동일 수정사항 반영
git checkout develop
git pull origin develop
git merge hotfix/99-session-timeout
git push origin develop
```

## 작업 흐름 예시

### 예시 1: AI 추천 알고리즘 구현

#### 1. Issue 생성

```
제목: [Feature] AI 추천 알고리즘 구현
라벨: enhancement
Issue #15
```

#### 2. 브랜치 생성

```bash
git checkout develop
git pull origin develop
git checkout -b feat/15-ai-draw-algorithm
```

#### 3. 커밋 작성

```bash
git commit -m "feat(draw): 코사인 유사도 계산 함수 구현"
git commit -m "feat(draw): AI 모드 뽑기 로직 추가"
git commit -m "test(draw): AI 추천 알고리즘 테스트 추가"
```

#### 4. Pull Request 생성

```
제목: feat(draw): AI 추천 알고리즘 구현
본문: Closes #15
브랜치: feat/15-ai-draw-algorithm → develop
```

이처럼 Issue → 브랜치 → 커밋 → PR 전체에서 동일한 타입(`feat`)과 Issue 번호(`#15`)를 사용합니다.

### 예시 2: 익명 댓글 버그 수정

#### 1. Issue 생성

```
제목: [Bug] 익명 댓글 순번 중복 발생
라벨: bug, priority:high
Issue #23
```

#### 2. 브랜치 생성

```bash
git checkout -b fix/23-anonymous-comment-bug
```

#### 3. 커밋 작성

```bash
git commit -m "fix(comment): get_anonymous_number 로직 개선"
git commit -m "test(comment): 익명 순번 중복 테스트 추가"
```

#### 4. Pull Request 생성

```
제목: fix(comment): 익명 댓글 순번 중복 버그 수정
본문: Resolves #23
브랜치: fix/23-anonymous-comment-bug → develop
```

## 참고사항

- 모든 작업 브랜치는 최신 `develop` 브랜치에서 분기합니다.
- 브랜치 이름만으로 작업 내용을 파악할 수 있어야 합니다.
- 하나의 브랜치는 하나의 목적만 가집니다.
- 장기간 유지되는 작업 브랜치는 주기적으로 `develop`을 병합하여 충돌을 최소화합니다.
- 브랜치 타입과 커밋 타입은 항상 일치시킵니다.
- 브랜치 생성 전 반드시 Issue를 먼저 생성합니다.
- Issue 번호를 브랜치 이름에 포함하여 작업 추적을 명확히 합니다.
- 실제 프로젝트의 기능과 구조를 반영한 브랜치명을 사용합니다.

## 브랜치 관리 팁

### 1. 브랜치 목록 확인

```bash
# 로컬 브랜치 목록
git branch

# 원격 브랜치 포함 전체 목록
git branch -a

# 병합된 브랜치 확인
git branch --merged develop
```

### 2. 오래된 브랜치 정리

```bash
# 병합된 로컬 브랜치 일괄 삭제
git branch --merged develop | grep -v "develop" | grep -v "main" | xargs git branch -d

# 원격에서 삭제된 브랜치 정리
git fetch --prune
```

### 3. 브랜치 간 이동

```bash
# 브랜치 전환
git checkout develop

# 이전 브랜치로 돌아가기
git checkout -
```
