# 브랜치 컨벤션 (Branch Convention)

## 1. 브랜치 네이밍 규칙 (Branch Naming Convention)

> 모든 브랜치는 다음 규칙에 따라 생성합니다.**`[브랜치타입]/[이슈번호]-[작업내용]`**예시: `feature/12-login`, `fix/34-login-bug`

🔹 **규칙**

- 브랜치 이름은 **소문자**로 작성
- 작업 내용은 **간결하고 명확하게**
- 이슈 번호가 있다면 반드시 포함
- 띄어쓰기 대신 `-` 사용

## 2. 브랜치 전략 (Branch Strategy)

> `master`의 권한은 팀장(강민규)에게 부여

- **브랜치 타입별 가이드**

| 브랜치 타입                      | 용도      | 설명                                        | 예시                        |
| --------------------------- | ------- | ----------------------------------------- | ------------------------- |
| `master`                    | 배포용     | 완성된 코드만 들어가는 메인 브랜치. 배포 시점마다 업데이트됨.       | `master`                  |
| `develop`                   | 개발용     | 각자 작업이 합쳐지는 통합 브랜치. 기능 테스트 후 master로 병합.  | `develop`                 |
| `feature`                   | 기능 개발   | 새로운 기능을 개발할 때 사용. develop에서 분기하여 작업 후 병합. | `feature/12-login`        |
| `fix`                       | 버그 수정   | 버그를 수정할 때 사용. develop 또는 master에서 분기.     | `fix/34-login-bug`        |
| `refactor`                  | 리팩토링용   | 코드 구조나 로직을 개선하는 브랜치 ( 기능 변화 없음).          | `refactor/27-auth-logic`, |
| `refactor/42-service-layer` |         |                                           |                           |
| `docs`                      | 문서수정용   | README, 위키, 컨벤션 등 문서 관련 변경 작업.            | `docs/10-readme`,         |
| `docs/19-convention`        |         |                                           |                           |
| `style`                     | 스타일 변경용 | 코드 포맷팅, 공백, 세미콜론 등 기능과 무관한 스타일 변경.        | `style/22-black-format`,  |
| `style/29-indent`           |         |                                           |                           |
| `test`                      | 테스트용    | 유닛테스트, 통합테스트 추가 및 수정용 브랜치.                | `test/17-user-api`,       |
| `test/23-ai-model`          |         |                                           |                           |
| `hotfix`                    | 긴급 수정   | 배포된 master에서 긴급 버그 수정 시 사용.               | `hotfix/56-deply-bug` ,   |
| `hotfix/87-api-crash`       |         |                                           |                           |

## 3. 브랜치 작업 흐름

🔹 **기본 작업 흐름**

1. `develop` 브랜치에서 최신 코드 pull
2. 새로운 브랜치 생성 (`feature/이슈번호-작업내용`)
3. 해당 브랜치에서 작업 진행
4. 작업 완료 후 커밋 및 푸시
5. Pull Request 생성하여 `develop`에 병합 요청
6. 코드 리뷰 후 승인되면 병합
7. 병합 완료된 브랜치는 삭제

🔹 **주의사항**

- 작업 전 반드시 `develop` 최신 버전 pull
- `master` 브랜치에는 직접 push 금지
- 큰 파일(`.csv`, `.pkl`, 이미지 등)은 `data/` 폴더에 두고 `.gitignore`에 추가
- `.env` 파일이나 개인정보 파일은 깃에 올리지 않기

# 커밋 컨벤션 (Commit Convention)

## 1. 커밋 메시지 형식

> **`[타입]: [작업 내용]`**예시: `feat: 로그인 기능 구현`, `fix: 크롤링 시 오류 수정`

## 2. 커밋 타입 (Commit Types)

| 타입         | 설명                     | 예시                     |
| ---------- | ---------------------- | ---------------------- |
| `feat`     | 새로운 기능 추가              | `feat: 로그인 기능 구현`      |
| `fix`      | 버그 수정                  | `fix: 크롤링 시 오류 수정`     |
| `docs`     | 문서 수정                  | `docs: README 업데이트`    |
| `refactor` | 코드 리팩터링                | `refactor: 중복 코드 정리`   |
| `style`    | 코드 스타일 변경 (포맷, 세미콜론 등) | `style: 불필요한 공백 제거`    |
| `test`     | 테스트 코드 추가              | `test: 유저 모델 테스트 추가`   |
| `design`   | CSS 등 사용자 UI 디자인 변경    | `design: css 수정`       |
| `chore`    | 기타 설정 변경               | `chore: .gitignore 수정` |

## 3. 커밋 메시지 작성 규칙

🔹 **규칙**

- 한 커밋에는 **하나의 목적**만 담기
- 커밋 메시지는 **명확하고 간결하게**
- 한글로 써도 괜찮지만 **일관성 유지**
- 동사 원형으로 시작 (추가, 수정, 삭제 등)

🔹 **커밋 메시지 예시**

**❌ 안 좋은 커밋 메시지**

- `feat: 로그인`
- `feat: 업데이트`
- `fix: 버그 수정`
- `fix: 최종 수정 완료`

**✅ 좋은 커밋 메시지**

- `feat: 로그인 기능 추가`
- `fix: 유저 데이터 페칭 오류 수정`
- `feat: add login functionality`
- `fix: correct user data fetching error`

## 4. 여러 줄 커밋 메시지 작성

🔹 **CLI에서 커밋 메시지 여러 줄로 작성하는 방법**

쌍따옴표를 닫지 말고 개행하며 작성 → 다 작성한 후에 쌍따옴표를 닫으면 작성 완료

```bash
git commit -m "feat: wandb 실험

- batch size 조정"
```

## 5. 커밋 전 체크리스트

- [ ] 커밋 메시지가 규칙에 맞나요?
- [ ] 한 커밋에 하나의 목적만 담겼나요?
- [ ] `.env` 파일 등은 깃에 올리지 않았나요?
- [ ] 최신 코드를 pull 받은 후 push 했나요?

# Pull Request (PR) 규칙

> PR은 "내 코드를 팀에 합치는 요청"

## 1. PR 제목 형식

```
[타입] 작업 내용
```

**예시**

```
[feat] 주식 상세 페이지 구현
[fix] 데이터 저장 오류 수정
```

## 2. PR 내용 작성 예시

```
### 작업 내용
- 주식 이름으로 검색 기능 추가
- 관련 주식 데이터 DB 저장 기능 구현

### 확인 요청
- 데이터 중복 저장 여부 한 번 확인 부탁드립니다!
```

## 3. PR 규칙

- 다른 팀원의 코드 리뷰 후 머지
- 작업 전 반드시 `develop` 최신 버전 pull
- PR 제목과 내용이 명확해야 함
- 관련 이슈 번호가 있다면 연결

---

# 코드 리뷰 규칙

> 서로의 코드를 리뷰하면서 품질 상승

- 리뷰어는 코드 확인 후 **approve** 또는 **수정 요청**
- 피드백은 간결하게 (`변수명 명확하면 좋겠어요!` 정도로)
- 리뷰는 하루 이내로 확인
- 리뷰 승인 후 브랜치는 삭제

# 파일 / 폴더 네이밍 규칙

| 구분         | 규칙             | 예시                          |
| ---------- | -------------- | --------------------------- |
| Python 파일  | 소문자 + 언더스코어(_) | `user_model.py`             |
| HTML / CSS | 소문자 + 대시(-)    | `user-profile.html`         |
| 폴더         | 소문자            | `templates`,`static`,`data` |

🔹 **규칙**

- 대문자 대신 소문자 사용
- 띄어쓰기 대신 `_` 또는 `-` 사용

# 전체 요약

| 항목         | 규칙 핵심                          |
| ---------- | ------------------------------ |

| **리뷰**     | 승인 후 병합                        |
